# bitig Security & Correctness Audit — v0.2.0

**Auditor:** Lead auditor, adversarially-verified pass
**Date:** 2026-06-03
**Scope:** 10 audit dimensions, 58 confirmed findings + 6 maintainer-judgment items
**Verdict:** **Not court-ready.** The forensic claims the README and recipes make are not yet backed by the integrity, calibration, and verification machinery a court-facing tool requires.

---

## 1. Executive summary

- **Overall health:** The core stylometry engine, Case model, and CLI are well-built and broadly correct. But every one of the *forensic-evidential* claims (tamper-evident sign-and-lock, ENFSI likelihood-ratio reporting, chain-of-custody enforcement) has at least one P0/P1 defect that defeats it. The gap between "marketed as court-grade" and "actually defensible" is the headline finding.
- **Single most important risk — the chain of custody does not close.** The sealed `case_state_hash` in `signed.json` is computed over a `case.json` that `save()` immediately overwrites, so it **can never be reproduced or verified** (`cases.py:613-656`, P0). Compounding this: *nothing in the tool ever verifies a signature* (no `bitig case verify`, default plugin is a no-op), the GUI lets an analyst run and sign **tampered evidence** (custody check is advisory only), and **Export PDF silently rewrites the signed report**, breaking its bound hash. Together these mean a "signed & locked" bitig case carries no real tamper-evidence today.
- **Second risk — the forensic LR is fiction.** The shipped `imposters_lr` recipe (mode='forensic', titled "authorship verification (LR)") **never computes an LR** — `GeneralImposters` emits an uncalibrated [0,1] win-fraction, and a fall-through can place the *candidate's name* in the headline numeric slot (P1). When an LR *is* present, the ENFSI verbal rung is classified from a **3-significant-figure-rounded display string**, flipping conclusions across category boundaries — including reversing the *direction* of evidence at LR≈1 (P1, reproduced end-to-end).
- **Method-level correctness gaps:** `ZetaEder`'s "Laplace smoothing" is an algebraic no-op identical to `ZetaClassic` (P1); `MTLD` returns **0.0 for maximally diverse text** (diversity inverted, P1); `EderDelta` weights by centroid variance rather than Eder (2015)'s frequency rank and is circular (P1); the `classify` CLI's default `loao` grouping is structurally degenerate (~0 accuracy, P1).
- **Severity counts:** 1 × P0, 25 × P1, 21 × P2, 11 × P3 (58 confirmed) + 6 needs-human-judgment. The P1 band is dominated by **security/integrity** (7) and **forensic/method correctness** (10).
- **Cross-cutting root cause:** Load-bearing forensic logic (the ENFSI ladder, headline-scalar derivation, latest-result loader) is **copy-pasted 2–4 times** with no parity test, and the **CI lint/type/PDF jobs never install the optional extras** that contain the GUI, report, and PDF code — so the entire court-facing surface ships effectively unguarded by CI.

## 3. P0 / P1 findings (full detail)

### P0 — Security & data integrity

#### P0.1 — Sealed `case_state_hash` is irreproducible by construction
**Files:** `src/bitig/cases.py:613-656` (also reported as `case-state-hash-unreproducible`, `cases.py:610-656`)
**What's wrong:** `mark_signed()` writes `payload['case_state_hash'] = self._case_state_hash()` (line 616) **while `record.signed` is still False**; `_case_state_hash()` (line 648) hashes the *live* `case.json` bytes. Then lines 628-635 flip `signed=True` and call `save()`, which rewrites `case.json` — permanently changing the very bytes that were hashed. The in-code comment ("so the payload's `case_state_hash` reflects pre-sign state") is self-defeating: that pre-sign `case.json` is never persisted. Reproduced empirically: sealed `3f24c5…` vs. recomputed-after-reload `d9464d…` → **mismatch**. `report/case_report.py:89` recomputes on the live (signed) case, so the **PDF/HTML footer disagrees with `signed.json`**. No test reloads a signed case and asserts equality, so the green suite never catches it.
**Why it matters:** The module's stated purpose (`signatures.py` docstring: "anyone re-running the same Case must produce the same hashes") is contradicted by its own anchor. This is the keystone of the entire chain-of-custody story, and it is broken before any verifier even runs. Every downstream integrity feature (signature verification, report binding) inherits this failure.
**Fix:** Compute `case_state_hash` over a **canonical, sign-invariant projection** of the record — `record.to_dict()` with the mutable `signed.*` / `runs` / `latest_run` fields excluded, plus `study.yaml` hash + sorted evidence hashes — so `_case_state_hash()` is identical pre- and post-sign **and** the report footer agrees with `signed.json`. Add a regression test: reload a signed case, assert `_case_state_hash() == signed.json['case_state_hash']` and that the rendered footer matches. Must be fixed before any `verify` feature can work.

### P1 — Security & data integrity

#### P1.1 — No signature verifier is wired anywhere; default plugin gives zero tamper-evidence
**File:** `src/bitig/signatures.py:70-81,150-151,169-189`
**What's wrong:** `DEFAULT_SIGNATURE_PLUGIN = NullSignaturePlugin()` (line 150) returns the payload unchanged — `signed.json` has no `signature` key under the default path. `verify_hmac_signature` (169-189) has **zero callers in `src/`** (only unit tests). There is no `bitig case verify` subcommand (`case_cmd.py:58-273` exposes only new/list/open/status/fork/sign). The only CLI integrity check, `status --verify`, calls `verify_custody()` which re-hashes evidence against the **mutable, unsigned `case.json`** and never touches `signed.json`'s `case_state_hash`/`report_html_hash`.
**Why it matters:** A signed report's `draft.html` or `signed.json` fields can be edited with nothing flagging it. The module docstring scopes crypto signing as a "future extension point," so this is an incomplete feature rather than a silently-broken control — but the tool must not be marketed as court-grade until it lands.
**Fix:** After fixing P0.1, add `bitig case verify <id>` (+ a GUI affordance) that recomputes `case_state_hash` + `report_html_hash` against disk and, when `signature_plugin_id != 'null'`, calls the matching verifier. Surface a prominent warning on `open()`/`status` when `signed.json` cannot be reproduced. Document that the Null plugin provides no tamper-evidence.

#### P1.2 — `add_evidence` `dest_name` allows writing outside the evidence dir (`../` traversal)
**File:** `src/bitig/cases.py:407-449`
**What's wrong:** `dest = role_dir / (dest_name or src.name)` (429), only `dest.exists()` is checked (430), then `shutil.copy2` (432). Reproduced: `dest_name='../../report/forged.html'` wrote a file into `<case>/report/` (alongside `signed.json` and `draft.html`) and registered the dotted path. `relative_to`/`is_relative_to` do **not** catch this without `.resolve()`.
**Why it matters:** Write-side traversal into the directory holding the signed artifacts. `dest_name` is public API.
**Fix:** Require `dest_name == Path(dest_name).name` (no separators, no `..`, not absolute); apply the same check to the `src.name` fallback; assert `dest.resolve().is_relative_to(role_dir.resolve())` before copy.

#### P1.3 — Unvalidated case `id` escapes the cases root (CLI + GUI)
**File:** `src/bitig/cases.py:300-363` (also `cases.py:317-320`)
**What's wrong:** `case_dir = root / id` (318) with no validation; `Case.load(root / case_id)` (357-363) trusts the path. Reproduced: `id='../evil_target/c'` created `case.json` **outside** the cases root, invisible to `list_cases`. Both entry points pass raw input: CLI `_resolve_case` (`case_cmd.py:43-45`), GUI `resolve_case` (`_helpers.py:22`) which feeds `case_id` straight from the `@ui.page('/case/{case_id}/...')` URL params (evidence/method/run/findings/report). *(Note: GUI URL exploitability depends on NiceGUI's `%2f` handling, not run-tested live; the create-side escape via typed/pasted id needs no caveat.)*
**Why it matters:** Both a create-side escape and a load-side traversal primitive against any directory containing `case.json`.
**Fix:** Enforce `^[A-Za-z0-9._-]+$`, reject `.`/`..`/separators/absolute paths, assert `(root / id).resolve().parent == root.resolve()`. Sanitize identically in CLI `_resolve_case` and GUI `resolve_case`.

#### P1.4 — Crafted `case.json` evidence path yields arbitrary-file read (and copy-out via fork)
**File:** `src/bitig/cases.py:356-363,459-474,658-728`
**What's wrong:** `Case.load` does `CaseRecord.from_dict(...)` with no path validation; `EvidenceEntry.from_dict` stores `path` verbatim. Reproduced: a hand-crafted `entry.path='../../secret.txt'` made `verify_custody` (`abs_path = self.root / entry.path`; `hash_file`) resolve and hash a file **outside** the case dir. `fork_case` (708-723) does `forked.add_evidence(source.root / entry.path, …)` → `shutil.copy2`, so a crafted source `case.json` **copies arbitrary host files** into a new case.
**Why it matters:** The product's portability/shareability premise means an attacker-supplied or corrupted `case.json` is a realistic threat.
**Fix:** Treat `case.json` as untrusted — in `EvidenceEntry.from_dict`/`Case.load`, reject absolute paths and any entry where `(root / path).resolve()` is not within `root.resolve()`, before the first `hash_file`/`copy2`.

#### P1.5 — Case can be signed with no report bound; GUI never regenerates report before signing
**File:** `src/bitig/cases.py:583-636`
**What's wrong:** `_report_html_hash` (658-662) returns `None` when `report/draft.html` is absent, with no error; `mark_signed` embeds it unchecked. Reproduced: signed a case with no `draft.html`, got `report_html_hash=null` with `signed=true` persisted. The GUI `_sign` handler (`report.py:76-84`) calls `mark_signed` directly and never calls `build_case_report` (that lives only in the separate `_export_pdf` button). `_case_state_hash` (644-656) also excludes `report_html_hash`/`draft.html` content, so even when present the report binding is independently editable.
**Why it matters:** A case can be permanently signed with no report, or bound to a stale draft. Compounds P0.1/P1.1.
**Fix:** In `mark_signed`, raise `CaseError` when `_report_html_hash()` is `None`; fold `report_html_hash` into `_case_state_hash`. In GUI `_sign`, call `build_case_report(case, format='html')` immediately before `mark_signed`.

#### P1.6 — `add_evidence(role='control')` copies file to disk before rejecting (orphan file)
**File:** `src/bitig/cases.py:427-447`
**What's wrong:** `'control'` passes role validation; `mkdir`, `copy2` (432), and `EvidenceEntry` construction all run **before** the rejection at 443-446. Reproduced: the orphan lands inside `evidence/control/` yet is invisible to `verify_custody()`/`compute_corpus_hash()`/`_case_state_hash()`. *(Blast radius limited: neither CLI nor GUI calls this path — they use `set_control_corpus()` — so reachable only via direct API misuse.)*
**Why it matters:** Violates the "failed mutation leaves the evidence tree byte-for-byte unchanged" invariant, and the stray file sits *inside* the integrity-tracked tree but outside the hash.
**Fix:** Move the `role=='control'` guard to the very top of `add_evidence`, before any filesystem mutation.

### P1 — Report rendering, PDF export & GUI integrity

#### P1.7 — Export PDF / `build_case_report` overwrites the signed `draft.html`, invalidating `report_html_hash`
**Files:** `src/bitig/report/case_report.py:62-75` and `src/bitig/gui/pages/case/report.py:67-101` (two reports of the same mechanism: `signed-draft-mutation-breaks-integrity` + `export-pdf-overwrites-signed-report`)
**What's wrong:** `build_case_report` **always** writes `report_dir/draft.html` (`case_report.py:65-67`) with **no signed guard** (contrast `_require_unsigned` at `cases.py:422/453/496/517/539`). `_build_context` stamps `date_iso=datetime.now(UTC)` and renders the SIGNED banner when `context.signed` is True, so a post-sign re-render produces a byte-different file that **fails hash verification yet displays as legitimately signed**. The GUI `_export_pdf` button (`report.py:91-93`) calls it with no signed guard. *(Correction to an over-statement in intake notes: the trigger is the Export-PDF button, **not** every page visit — `case_report_page` renders the body with NiceGUI widgets and never touches `draft.html`. And no in-tool path currently re-checks `report_html_hash`, so impact today is on external/HMAC verification.)*
**Why it matters:** A sign-and-lock tool that rewrites the locked artifact is the canonical tamper-evidence hole.
**Fix:** Guard the write — when `case.record.signed`, either raise `ReportRendererError` ("Case is signed; fork it for further work") or render the PDF from the existing frozen `draft.html`. Best: have `mark_signed` snapshot to an immutable `report/signed.html` that export never rewrites. Disable the Export-PDF button on signed cases (mirror `sign_btn`). Add the case-verify step (P1.1) so the tool detects this itself.

#### P1.8 — Figure `<img src>` paths are off-by-one vs. `base_url`; every figure 404s in PDF and draft HTML
**File:** `src/bitig/report/case_report.py:258-268`
**What's wrong:** `_list_figure_paths` (264) computes `fig.relative_to(case.report_dir.parent)` → root-relative `runs/<ts>/verify/lr_plot.png`, but `build_case_report:74` passes `base_url=case.report_dir` (=`<root>/report`). Reproduced: resolved against `report_dir` → does not exist; against root → exists. The same mismatch breaks a browser opening `draft.html`. The in-code comment is self-contradictory. **Zero coverage** — the only PDF test is skipped (WeasyPrint not installed).
**Why it matters:** Every evidentiary LR/reliability plot is **silently absent** from the exported court report.
**Fix:** Set `base_url=case.root` (or compute `fig.relative_to(case.report_dir)` → `../runs/...`). Add a WeasyPrint-gated test asserting figure bytes are embedded; install `bitig[reports]` in at least one CI job.

#### P1.9 — Chain-of-custody mismatch is advisory only in the GUI; analyst can run and sign tampered evidence
**File:** `src/bitig/gui/pages/case/evidence.py:31-68`
**What's wrong:** `verify_custody()`'s own docstring (`cases.py:462-463`) says the Evidence step uses it to "flip cards red and **block step 4+**." The GUI computes custody (32) and renders a red banner (33-44) but gates "Next: Method" solely on `bool(...questioned or ...known)` (62-68) — custody is never consulted. `run.py:57-58` guards only on `signed`. The stepper pills navigate unconditionally (`case_layout.py:130`). *(The CLI **does** enforce: `status --verify` raises `typer.Exit(code=2)` — so a scripted gate exists; the spec-mandated GUI block does not.)*
**Why it matters:** A case with tampered evidence shows a banner yet can be carried Method→Run→Sign into a signed report over compromised evidence. The difference between "detected" and "prevented."
**Fix:** Gate "Next: Method" on `not custody` (already computed at line 32). Add a hard `verify_custody()` re-check inside `run.py` `execute()` before `run_study`, aborting on mismatch; optionally block `_sign` likewise. Mirror the CLI's exit-2 semantics.

### P1 — Forensic-linguistic correctness (LR / verification)

#### P1.10 — Shipped forensic recipe never computes an LR; report headline can show the candidate's NAME in the score slot
**Files:** `methods/imposters.py:197-204`; `gui/pages/case/_helpers.py:80-87`; `report/case_report.py:160-175,92-93,310-331`; `runner.py:229-246`
**What's wrong:** Reproduced end-to-end. (1) `GeneralImposters.Result.values = {'candidate','imposters','threshold','scores'}` — **no `lr`/`log_lr`**, win-fraction never calibrated. (2) `runner.py` `kind=='verify'` returns the GI Result unchanged; only the `classify` branch ever adds calibration. The shipped `_IMPOSTERS_LR` recipe (`recipes.py:86-101`, mode='forensic') uses `kind='verify'`. (3) `headline_scalars` returns `[('score','suspect_A')]` — the **candidate name** — because dict order puts `'candidate'` first and `next(iter(v.values()))` grabs it. (4) Because the label is `'score'` not `'LR'` and `_maybe_float('suspect_A')=None`, the entire Likelihood-ratio section and ENFSI ladder are **silently dropped** from a report titled with LR framing. `CalibratedScorer`/`log_lr_from_probs` exist in `forensic/lr.py` but are not wired in.
**Why it matters:** The central forensic-correctness defect — a court-facing "authorship verification (LR)" workflow produces no LR.
**Fix:** (a) Never let a non-numeric value reach the headline/LR slot — replace `next(iter(v.values()))` in `_helpers.py:87` and `case_report.py:170-174` with a numeric (e.g. mean of `v['scores']`, labeled "GI score"). (b) Decide the story: wire GI scores through `CalibratedScorer.log_lr` (storing real `lr`/`log_lr` when a calibration set is configured), **or** relabel the recipe/report as "GI verification score" and drop the LR framing/ladder until calibration exists. Do not present an uncalibrated [0,1] win-fraction as an LR.

#### P1.11 — Forensic report highlights the WRONG ENFSI rung (classified from a rounded LR display string)
**Files:** `src/bitig/report/case_report.py:92-114, 297-331` (reported three times: `verbal-rung-round-then-classify` + `verbal-rung-from-rounded-string` + the GUI mirror in `gui-report-page-untested`, `gui/pages/case/report.py:127-128,142`)
**What's wrong:** `case_report.py:93` sets `lr_value = scalars[0].value` — the `_fmt()` **string** (`%.3g`/`%.2e`), which rounds `9.999→'10'`, `99.95→'100'`, `999.6→'1e+03'`, `9995→'1e+04'`, `0.9999→'1'`. Line 112 passes that string into `_verbal_rung_for_value` which re-parses with `float()` and classifies on hardcoded decade cutoffs. Reproduced with the project's own functions: `9.999→'moderate'` (correct: weak); `99.95→'strong'` (correct: moderate); `999.6→'very strong'` (correct: strong); `9995→'extremely'` (correct: very strong); and most consequentially **`0.9999→'weak support'` (correct: 'no support') — reversing the direction of evidence** from defence to prosecution. `forensic.html.j2:84-95` highlights this as the active ENFSI rung — the load-bearing court conclusion. The **GUI live preview has the identical defect** (`report.py:127-128,142` feeds the `_fmt` string through `_maybe_float` into `lr_verbal_rung`; 4/4 mismatches reproduced). The only test uses `LR=250.0` (mid-rung), so CI never exercises a boundary.
**Why it matters:** A display-formatting artifact flips the ENFSI category — indefensible in a court tool.
**Fix:** Classify the rung from the **raw float** `result.values['lr']` (and `log_lr`→LR), never the `_fmt` string — in both `case_report.py` `_build_context` and `gui/pages/case/report.py`. Add boundary parity tests at `0.999, 1.0, 9.999, 10.0, 99.95, 100.0, 999.6, 1000.0, 9995, 10000.0` asserting equal rungs across report + GUI. Consolidate onto one classifier (see P2 theme below).

### P1 — Stylometry method & feature correctness

#### P1.12 — `ZetaEder` "Laplace smoothing" is an algebraic no-op, identical to `ZetaClassic`
**File:** `src/bitig/methods/zeta.py:131-139`
**What's wrong:** Returns `(proportion_a[w] + 0.5)/1.0 - (proportion_b[w] + 0.5)/1.0`. The `+0.5` cancels, `/1.0` is identity → exactly `ZetaClassic._score` (`zeta.py:128`). Reproduced: classic and eder agree to 1.1e-16 (FP noise). Worse, the smoothing is applied to **proportions, not counts** — even a non-cancelling constant would not be valid Laplace smoothing. The test only asserts `len(res.tables) == 2`. `ZetaEder` is publicly exported and selectable as runner variant `'eder'`.
**Why it matters:** A documented, user-selectable method silently does nothing different from another.
**Fix:** Implement real add-k smoothing on document-occurrence **counts**: `prop_x(w) = (count_x(w)+k)/(n_x+2k)`, then difference. Add a regression test asserting Eder ≠ Classic on a corpus with a zero-count word, pinning one expected smoothed value.

#### P1.13 — `MTLD` returns 0.0 for maximally diverse text (diversity ordering inverted)
**File:** `src/bitig/features/lexical_diversity.py:74-99`
**What's wrong:** Reproduced: `_mtld(['w0'..'w99'])` (all unique, TTR=1.0) returns **0.0**, while a highly repetitive text returns 5.0 — maximal diversity scores *below* minimal. Root cause (91-95): an all-unique trailing segment gives `partial=(1-1.0)/(1-0.72)=0`, `factor_count` stays 0, and line 95 returns `0.0`. McCarthy (2005) never assigns 0 to maximally diverse text. Affects any document whose forward or backward pass never crosses threshold (the two are averaged at line 99). `mtld` is a selectable index.
**Why it matters:** Directly harms short/high-diversity questioned texts common in forensic stylometry — the score is backwards exactly where it matters.
**Fix:** Per McCarthy (2005), a never-crossed-threshold segment's partial factor must reflect remaining distance, and a text completing zero factors should be one incomplete factor of length N (large MTLD), not 0. Add monotonicity tests.

#### P1.14 — `EderDelta` weights by across-centroid variance, not Eder (2015) frequency rank; circular and self-contradictory
**File:** `src/bitig/methods/delta/eder.py:11-34`
**What's wrong:** Code ranks features by `stacked.var(axis=0)` (across-centroid variance) then weights `(n-rank)/n`. Verified the highest-variance column gets weight 1.0 — driven by centroid spread, not frequency. The docstring asserts both "most frequent features contribute most" *and* "mean absolute z-score" — neither is true, so it self-contradicts. **Circular:** weights derive from `self.centroids_`, the very class means it then classifies against. `MFWExtractor` already sorts columns by descending frequency (`mfw.py:87`), so a genuine Eder weight is simply `np.arange(n,0,-1)/n`.
**Why it matters:** A named, exported metric does not implement the method it cites and fits to the classes it scores.
**Fix:** Replace centroid-variance ranking with frequency-position weights `np.arange(n,0,-1)/n` in the extractor's frequency order. Do not derive weights from the centroids. If a discriminativeness-weighted delta is wanted, expose it under a distinct, accurately named class. Rewrite the docstring; pin the weight vector in a test.

#### P1.15 — Bayesian attributor silently accepts z-scored features, destroying the rate model
**File:** `src/bitig/methods/bayesian.py:84-107`
**What's wrong:** Reproduced end-to-end: fed negative (z-scored) values, no error; `np.clip(rates, 1e-12, 1.0)` (line 100) silently clamps negatives and >1 values, yielding confident but **scientifically invalid** finite-label predictions. The docstring warns but enforces nothing. `runner.py:301-305` passes features unchecked, and `MFWExtractor` defaults `scale='zscore'` (`mfw.py:57`) — only the standalone CLI hardcodes `scale='none'`. So a study-config Bayesian method pointed at a z-score feature produces silent garbage.
**Why it matters:** Confident, invalid attributions from a method whose whole premise is non-negative rates.
**Fix:** In `fit()`, raise on `(counts < 0).any()` with a clear message; guard `smoothed.sum() > 0`. In the runner's Bayesian branch, assert the feature was built with `scale ∈ {none, l1}` (or rebuild it).

#### P1.16 — `classify` CLI default `cv_kind='loao'` with `groups=labels` is structurally degenerate (~0 accuracy)
**File:** `src/bitig/cli/classify_cmd.py:24-42`
**What's wrong:** Default `cv_kind='loao'` (24) with `groups_from=y` (39) — the author label array, same as the target. `LeaveOneGroupOut` holds out every doc of an author per fold, so the held-out author is never in training. Reproduced: accuracy = **0.0 exactly**. There's no CLI flag for an independent grouping column. *(The study-config path correctly requires an explicit `cv.groups_from`; the defect is CLI-specific.)*
**Why it matters:** The default invocation `bitig classify corpus -m meta.csv` reports a meaningless ~0 accuracy.
**Fix:** Default to `cv_kind='stratified'`, or add `--groups-by` (validated ≠ `group_by`). Raise a clear error if `groups_from` is element-wise identical to `y`.

#### P1.17 — `hdbscan` and `umap-learn` are hard runtime deps but only lazily imported
**File:** `pyproject.toml:43-44`

### P1 — Test coverage & quality

#### P1.18 — GUI run orchestration `execute()` is entirely untested; can mark a case "run complete" when every method failed
**File:** `src/bitig/gui/pages/case/run.py:29-114`
**What's wrong:** Zero test references to `case_run_page`/`execute`/`perform_run`. The all-failed risk is real: `runner.py:131-167` catches every per-method exception, writes `error.txt`, continues, and `run_study` **unconditionally returns** `run_dir`. Back in the GUI, `register_run` only checks `run_dir.is_dir()`, lists the failed method dirs, pushes "✓ run complete," sets status 'done' → Findings/Report unlock over a run where every method errored. `_new_run_id` formats to **second resolution**, and `run_study` reuses the dir, so two runs in one second silently overwrite.
**Why it matters:** Court workflow can present a "complete" run that wholly failed.
**Fix:** Extract `execute()`'s body into a pure `perform_run(case)->RunOutcome` distinguishing all-succeeded / partial / all-failed; render from that. Unit-test the signed guard, all-failed (must NOT unlock Findings), and run-id collision. Add a runner integration test where a method raises. Use sub-second or counter-suffixed run ids.

#### P1.19 — WeasyPrint PDF export (the court deliverable) is never exercised by any test, in any environment

## 5. Open methodology / design questions (maintainer decisions)

These six are correct-as-coded but require a domain or design call:

1. **[Forensic — P2] Should the LR verbal ladder be symmetric?** Raw-LR ladders collapse all `0<LR<1` to "no support" (`_helpers.py:126-143`, `case_report.py:310-331`), hiding exculpatory (defence-favouring) evidence — `LR=0.001` is *strong support for Hₐ*, not "no support." The ENFSI/Nordgaard scale is symmetric about log₁₀-LR=0. **Decision:** when a real calibrated LR is wired (P1.10), report `LR∈[0.1,1)` as "weak support for Hₐ," etc. by log₁₀ magnitude, reserving a narrow band around LR=1 for "inconclusive." Also reconcile which recommendation governs — the `LR_LADDER` comment cites *Marquis et al. ENFSI 2016* while `forensic_lr.j2` cites *Nordgaard 2012 / ENFSI 2015*. **This is the load-bearing forensic-scale consistency question** and should be resolved together with P1.10 and P1.11's centralization.

2. **[GUI — P2] Threading model for `run_study`.** It's CPU-bound but dispatched via `run.io_bound` (a thread), giving illusory progress and potential event-loop stutter (`run.py:74-79`). The `cpu_bound` switch isn't free — it requires `run_study` + a `SpacyPipeline` (`en_core_web_trf`) to pickle cleanly under macOS spawn. **Decision:** spike `cpu_bound` behind a spawn smoke test, or keep `io_bound` and stream honest progress. Same pattern recurs in legacy `pages/run.py:52` and `forensic.py`.

3. **[GUI — P3] Module-global `GuiState` singleton shared across all clients/tabs** (`state.py:31-44`). Safe under the single-window native deployment (127.0.0.1, PyWebView). **Decision:** either document/enforce single-client ("--no-native is dev-only"), or migrate user-scoped fields to `app.storage.tab/user`.

4. **[Release — P2] `publish-pypi` requires a `pypi` GitHub environment + PyPI Trusted Publisher that may not exist** (`publish.yml:6-14,64-89`). Not statically verifiable. **Decision/checklist before the next tag:** confirm/create the `pypi` environment; register the Trusted Publisher (owner=fatihbozdag, repo=bitig, workflow=publish.yml, env=pypi); dry-run via `workflow_dispatch` or TestPyPI first.

5. **[Packaging — P3] `Development Status :: 2 - Pre-Alpha` despite two releases and court-readiness marketing** (`pyproject.toml:15`). **Decision:** bump to Beta/Alpha to match the maturity claims, or soften the README's "court-facing" framing.

6. **[API — P3] Entire Forensic Lab subsystem (Case/recipes/signatures/case_report) absent from top-level `__all__`** (`__init__.py:61-152`). Likely deliberate (CLI/GUI-first; `build_forensic_report` is also submodule-only). **Decision:** document the submodule-only convention, or promote the stable Case API to the root (and fix the pre-existing `build_report`-exported / `build_forensic_report`-not-exported inconsistency in the same pass).

## 6. Cross-cutting themes

1. **Forensic logic is copy-pasted, not shared.** The ENFSI ladder exists in **4 places** (P1.11/P2 ladder finding), headline-scalar derivation in 2, the latest-result loader in 2, and `_fmt` verbatim in 2. None have parity tests. The verbal-rung bug (P1.11) is the *already-realized* consequence of this pattern; the others are drift waiting to happen. **One shared, non-GUI classifier + one ladder constant** (in `bitig.forensic` or a small `bitig.report` helper, importable by both `report/` and `gui/`) collapses P1.11 + 4 P2s at once.

2. **The signed chain of custody is hollow end-to-end.** P0.1 (irreproducible hash) + P1.1 (no verifier, Null default) + P1.5 (sign with no/stale report, hash excludes report) + P1.7 (export rewrites the locked file) + P1.9 (GUI runs/signs tampered evidence) are **one coherent failure**: nothing the tool produces under "sign & lock" can be independently verified, and several paths actively invalidate it. Fix as a single workstream, in the order P0.1 → P1.5 → P1.1 → P1.7 → P1.9.

3. **CI never installs the optional extras**, so the entire court-facing surface — GUI (`nicegui`), report (`weasyprint`), interactive (`plotly`), Bayesian (`pymc`) — ships with **vacuous lint/type coverage** (`ci.yml:23-26`) and **zero PDF happy-path coverage** (P1.19). The untested-orchestration findings (P1.18, the P2/P3 GUI/report-page gaps) are downstream of this. A single lint/type/test job with the relevant extras installed (with the `WindowProxy` typing handled correctly, *not* by "fixing" the correct awaits) closes most of it.

4. **Filesystem inputs are trusted that shouldn't be.** Three path-traversal vectors (P1.2 write, P1.3 id, P1.4 crafted `case.json` read/copy-out) plus the CWD-relative DocBin cache (P2) all stem from treating operator/disk input as safe. For a tool whose premise is *sharing portable cases*, `case.json` and `dest_name` are untrusted boundaries. Add one `_safe_component`/`_within_root` helper and apply uniformly.

5. **The release pipeline has multiple latent failure modes** that won't bite a standard final-version tag but will surprise on the next non-trivial release: the `sed` regex (P2), the dispatch bypass (P2), the unconfigured Trusted Publisher (judgment #4), and the stale `CITATION.cff` (P2). Plus non-atomic state writes (P2) threaten the integrity root on any interrupted save.

- **Event-loop starvation magnitude (judgment #2).** Whether `io_bound` `run_study` materially stutters the UI depends on GIL-release behavior of the BLAS/spaCy-transformer C extensions during a real `en_core_web_trf` run — needs a timed live run, and the `cpu_bound`+macOS-spawn pickling of `SpacyPipeline` needs a smoke test.
- **PyPI Trusted Publisher / `pypi` environment existence (judgment #4).** Out-of-repo state; verify before the next tag.
- **Forensic-scale literature reconciliation (judgment #1).** Whether the canonical verbal scale should be raw-LR 5-rung (Marquis et al. ENFSI 2016) or log₁₀ 6-band (Nordgaard 2012 / ENFSI 2015), and how to express two-sided (defence-favouring) support, is a domain call requiring the maintainer's reading of the standards — not resolvable by code inspection.
- **Whether resubstitution accuracy is ever acceptable in the delta study branch (P2).** A methodology call on the diagnostic-vs-attribution distinction.
- **Calibration validity (tied to P1.10).** Even after wiring `CalibratedScorer.log_lr`, whether the resulting LRs are *well-calibrated* on real forensic data (Cllr, ECE, reliability diagrams against held-out sets) is an empirical question for the runtime pass, not static audit.
- **Which README test count is correct (P3)** — `pytest -q` was not run; one of 493/417 is stale.

**Bottom line for the maintainer:** the stylometry library is solid, but every forensic-evidential guarantee bitig advertises has a confirmed P0/P1 defect. Treat items 1–3 of the remediation order as blocking before any further "court-grade" framing, and route the runtime/calibration questions above into your planned forensic analysis pass.
